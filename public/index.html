Motor en el mismo navegador (client-side): Algoritmo A = DES-EDE + decimalización (CVE). Algoritmo B = HMAC-SHA256 + tiempo (CDT). Solo datos ficticios.

Código de Verificación Estático (CVE)

Comportamiento 3DES: CED = CVKA‖CVKB (32 hex). Motor en el navegador.

Código Dinámico Temporizado (CDT)

Alternativa más segura: HMAC + ventana de 30s → 6 dígitos que rotan. Más difícil de precomputar.

Stack del sandbox (EduCard Lab)

  • Frontend: HTML + CSS + JS (vanilla)
  • Backend local: Node.js http en 127.0.0.1:8787
  • Cálculo principal: motor.js en el navegador (mismo patrón client-side)
  • Algoritmo A: DES-EDE + decimalización · Algoritmo B: HMAC-SHA256 (Web Crypto)
  • Paquete práctica: paquete_practica.txt · automatizar: npm run automatizar

Hallazgo de arquitectura (sitio de referencia público)

  • Shell HTML + menú lateral; cada herramienta es un div (p. ej. #cvv-calculator) cargado por JS
  • Stack visible: Bootstrap, jQuery 3.7.0, /js/common.js, /js/main.js
  • Su propia política: cálculo client-side en el navegador; no envían los datos del formulario a un servidor de cálculo
  • Implicación: automatizar ese sitio = controlar un navegador, no “hablar con un API de CVV”
  • Mundo real (emisores): claves en HSM, PCI DSS, sin calculadoras públicas ni claves en el cliente

Uso exclusivamente académico · datos 100% ficticios · no procesa transacciones reales
